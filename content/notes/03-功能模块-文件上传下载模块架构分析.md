## 二、核心设计模式

### 2.1 策略模式 + 工厂模式

**类结构**：

```
IOssService (接口)
     │
     ▼
AbstractOssService (抽象类 - 公共逻辑)
     │
     ├── MinioServiceImpl (MinIO实现)
     ├── (未来) AliyunOssServiceImpl (阿里云OSS实现)
     └── (未来) AwsS3ServiceImpl (AWS S3实现)
```

**工厂代码** (`OssServiceFactory.java:37-40`)：

```java
public IOssService getOssService(OssTypeEnum ossType) {
    return ossServices.stream()
        .filter(service -> service.getType().equals(ossType))
        .findFirst()
        .orElseThrow(()->new BizException(BizRespCode.ERR_2300));
}
```

**设计目的**：

| 目的 | 说明 |
|------|------|
| 解耦 | 上层Service不关心底层用什么存储 |
| 可扩展 | 换OSS只需新增一个实现类，不改业务代码 |
| 统一接口 | 无论MinIO/阿里云OSS/AWS S3，调用方式一致 |

### 2.2 模板方法模式

`AbstractOssService` 封装公共流程，子类只需实现具体存储逻辑：

| 方法 | 职责 |
|------|------|
| `getMimeTypeByFileName()` | 根据文件后缀获取MIME类型 |
| `getFileKey()` | 生成文件存储路径（日期/UUID.后缀）|
| `checkSize()` | 文件大小校验 |
| `checkFileType()` | 文件类型白名单校验 |
| `setCacheResponseAndValid()` | HTTP缓存处理 |

### 3.1 流程图

public FileStoreVO upload(MultipartFile file) {
    // 1. 上传到OSS
    OssUploadDTO dto = ossServiceFactory.getDefaultOssService().uploadByMul(file);
    // 2. 保存元数据到数据库
    SysFile entity = new SysFile();
    entity.setName(dto.getFileName());
    entity.setObjectName(dto.getObjectName());
    entity.setByteSize(dto.getFileSize());
    entity.setFileType(dto.getFileType());
    entity.setFileUrl(dto.getFileUrl());
    this.save(entity);
    // 3. 返回VO
    return FileStoreVO.builder()
            .id(entity.getId())
            .objectName(entity.getObjectName())
            .fileName(entity.getName())
            .fileSize(entity.getByteSize())
            .fileType(entity.getFileType())
            .fileUrl(entity.getFileUrl())
            .build();
}

**OSS实现层** (`MinioServiceImpl.java:82-114`)：

```java
@Override
public OssUploadDTO uploadByMul(MultipartFile inputFile) {
    // 1. 生成文件存储key
    String fileKey = getFileKey(inputFile.getOriginalFilename());
    // 2. 校验大小和类型
    checkSize(inputFile);
    checkFileType(inputFile);
    // 3. 上传到MinIO
    @Cleanup final InputStream inputStream = inputFile.getInputStream();
    final PutObjectArgs putObjectArgs = new PutObjectArgs.Builder()
            .contentType(getMimeTypeByFileName(fileKey))
            .bucket(minioConfig.getBucket())
            .object(fileKey)
            .stream(inputStream, inputFile.getSize(), -1)
            .build();
    minioClient.putObject(putObjectArgs);
    // 4. 返回上传信息
    return OssUploadDTO.builder()
            .fileName(inputFile.getOriginalFilename())
            .fileSize(inputFile.getSize())
            .objectName("/".concat(ossConfig.getMinio().getBucket().concat("/").concat(fileKey)))
            .fileType(FileUtil.extName(inputFile.getOriginalFilename()))
            .fileUrl(getFileUrl(fileKey))
            .build();
}
```

### 4.2 HTTP缓存设计

`AbstractOssService.java:452-459`：

```java
public boolean setCacheResponseAndValid(String objectName, HttpServletResponse response) {
    // 设置缓存过期时间1天
    response.setHeader(HttpHeaders.CACHE_CONTROL, CacheControl.maxAge(Duration.ofDays(1)).toString());
    response.setHeader(HttpHeaders.EXPIRES, String.valueOf(System.currentTimeMillis() + TimeUnit.DAYS.toMillis(1)));
    // 设置ETag
    response.setHeader(HttpHeaders.ETAG, "soil-file-key-" + objectName);
    // 判断是否命中缓存
    HttpServletRequest request = ((ServletRequestAttributes) (RequestContextHolder.currentRequestAttributes())).getRequest();
    String ifNoneMatch = request.getHeader(HttpHeaders.IF_NONE_MATCH);
    return StringUtils.equals(ifNoneMatch, "soil-file-key-" + objectName);
}
```

**缓存流程**：

```
第一次请求：
  服务器返回文件 + ETag + Cache-Control

第二次请求（缓存未过期）：
  浏览器携带 If-None-Match: soil-file-key-xxx
  服务器判断ETag匹配 → 返回 304 Not Modified（不传输文件内容）
```

## 六、文件关联设计

### 6.1 业务场景

一个工单可以上传多个附件，但附件是先上传后关联的：

```
1. 用户上传附件 → 获得文件ID
2. 用户提交工单 → 工单创建成功，获得工单ID
3. 建立关联 → 更新 sys_file.model_id = 工单ID
```

### 6.2 关联方法

`SysFileServiceImpl.java:173-196`：

```java
@Override
public void associate(String modelId, String modelCode, List<String> fileIds) {
    // 1. 查询旧的关联文件
    List<SysFile> oldFileList = list(new LambdaQueryWrapper<SysFile>()
            .eq(SysFile::getModelId, modelId)
            .eq(StringUtils.isNotBlank(modelCode), SysFile::getModelCode, modelCode));
    Set<String> oldFileIds = oldFileList.stream().map(SysFile::getId).collect(Collectors.toSet());

    // 2. 如果新旧一致，直接返回
    if (oldFileIds.size() == fileIds.size() && oldFileIds.containsAll(fileIds)) {
        return;
    }

    // 3. 先解除旧的关联
    new LambdaUpdateChainWrapper<>(getBaseMapper())
            .set(SysFile::getModelId, null)
            .set(SysFile::getModelCode, null)
            .eq(SysFile::getModelId, modelId)
            .eq(StringUtils.isNotBlank(modelCode), SysFile::getModelCode, modelCode)
            .update();

    // 4. 建立新的关联
    if (CollUtil.isNotEmpty(fileIds)) {
        new LambdaUpdateChainWrapper<>(getBaseMapper())
                .set(SysFile::getModelId, modelId)
                .set(SysFile::getModelCode, modelCode)
                .in(SysFile::getId, fileIds)
                .update();
    }
}
```

**设计优点**：
- 文件上传和业务绑定分离
- 支持文件预上传，后绑定业务
- 避免孤儿文件

## 七、安全设计

### 7.1 文件大小限制

```java
public void checkSize(MultipartFile inputFile) {
    if (ossConfig.getMaxSize() == null || ossConfig.getMaxSize() <= 0) {
        return;
    }
    if (inputFile.getSize() > ossConfig.getMaxSize()) {
        log.error("文件大小超过限制, 传入文件字节数为:[{}], 文件最大字节数为:[{}]",
            inputFile.getSize(), ossConfig.getMaxSize());
        throw new BizException(BizRespCode.ERR_2302);
    }
}
```

### 7.2 文件类型白名单

`AbstractOssService.java:427-444`：

```java
public void checkFileType(MultipartFile inputFile) {
    if (StringUtils.isBlank(ossConfig.getAllowFileType())) {
        return;
    }
    String extension = FileUtil.getSuffix(inputFile.getOriginalFilename());
    String[] fileTypes = ossConfig.getAllowFileType().split(",");
    if (ArrayUtils.isNotEmpty(fileTypes)) {
        boolean valid = ArrayUtil.containsIgnoreCase(fileTypes, extension);
        if (!valid) {
            log.error("文件类型不正确, 传入类型为:[{}], 限制类型为:[{}]",
                extension, ossConfig.getAllowFileType());
            throw new BizException(BizRespCode.ERR_2303);
        }
    }
}
```

## 十、面试考点

### 10.1 如何回答"介绍一下文件上传模块"

**参考答案**：

> 我们项目的文件上传模块采用了策略模式+工厂模式的架构设计。整个模块分为三层：
>
> 1. **Controller层**：负责HTTP入口，接收 MultipartFile
> 2. **Service层**：负责业务逻辑，包括数据库记录、文件关联
> 3. **OSS层**：负责具体存储实现，通过接口抽象，支持切换不同存储方案
>
> 关键设计点：
> - **数据库记录元数据**：不是为了重复存储，而是管理文件归属、关联业务、权限控制
> - **modelId + modelCode**：实现文件与业务的灵活绑定，支持预上传后关联
> - **HTTP缓存**：ETag + Cache-Control，304减少带宽消耗
> - **安全校验**：文件大小限制 + 类型白名单

### 10.2 追问：为什么要用数据库记录文件信息？

**回答要点**：
1. 文件归属追踪（谁上传的）
2. 业务关联（modelId + modelCode）
3. 元数据管理（统一管理文件名、大小、类型）
4. 权限控制基础
5. 业务统计支持

### 10.3 追问：如果要换OSS怎么办？

**回答要点**：
1. 新建一个实现类，实现 `IOssService` 接口
2. 使用 `@ConditionalOnProperty` 条件装配
3. 修改配置文件中的 `defaultOssType`
4. 业务代码无需修改

### 10.4 追问：大文件上传怎么做？

**回答要点**：
1. **分片上传**：将大文件分成多个小块上传
2. **断点续传**：记录已上传分片，中断后继续
3. **秒传**：计算MD5，已存在则跳过
4. **客户端直传**：预签名URL，客户端直接上传到OSS，不经过后端
