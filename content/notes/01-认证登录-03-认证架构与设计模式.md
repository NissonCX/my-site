## 一、登录流程

```
1. 登录请求
       │
       ▼
2. AuthServiceFactory 路由
       │
       ▼
3. *AuthServiceImpl 实现具体登录逻辑
       │
       ▼
4. AbstractAuthTemplate.generateToken()
       │
       ├─ StpUtil.login(userId)      → 生成 JWT
       ├─ UserContext.setUserToken()    → 存入 Session (Redis)
       │
       ▼
5. 返回 accessToken 给客户端
```

## 五、设计模式

### 5.1 抽象模板模式

```
┌─────────────────────────────────────────────────────────────┐
│                    抽象模板模式                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  AbstractAuthTemplate (抽象类)                              │
│      │                                                      │
│      ├─ 公用逻辑：generateToken() 【固定不变】              │
│      │                                                      │
│      └─ 抽象方法：getOrCreateUser() 【子类实现】            │
│              │                                              │
│              ▼                                              │
│      ┌─────────┬─────────┬─────────┬─────────┐             │
│      ▼         ▼         ▼         ▼         ▼             │
│    YKB       YKZ       系统      H5       微信             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**模板方法的作用**：
- 将"生成 Token"这个固定逻辑封装在父类
- 子类只需实现"获取或创建用户"这个变化的部分
- 避免重复代码，保证核心流程一致

**为什么这样设计**：
- 各登录方式的核心流程一致（生成 JWT、存 Session）
- 但获取用户的逻辑不同（YKZ 调用阿里 SSO，微信调用微信 API）
- 抽象模板模式将"不变"和"变化"分离

### 5.2 工厂模式

```
┌─────────────────────────────────────────────────────────────┐
│                     工厂模式                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  AuthServiceFactory                                         │
│      │                                                      │
│      ├─ 注入：List<IAuthService> (所有实现)                 │
│      │                                                      │
│      └─ 路由：根据 authType 匹配                            │
│              │                                              │
│              ▼                                              │
│         具体实现类                                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**工厂的作用**：
- 根据请求的 `authType`（YKB/YKZ/WE_CHAT 等）找到对应实现
- 解耦 Controller 和具体实现
- 新增登录方式只需新增实现类，不需要改工厂

**Spring 的自动注入**：
- Spring 自动将所有 `IAuthService` 实现注入到 List 中
- 工厂只需遍历 List，根据 authType 筛选

## 六、为什么用这两种设计模式？

### 6.1 抽象模板模式的必要性

| 问题 | 解决方案 |
|------|---------|
| 各登录方式流程相似，但有差异 | 公共逻辑在父类，差异逻辑在子类 |
| 避免重复代码 | generateToken() 只写一次 |
| 保证流程一致性 | 所有子类都遵循相同流程 |

### 6.2 工厂模式的必要性

| 问题 | 解决方案 |
|------|---------|
| Controller 不知道该调用哪个实现 | 工厂根据 authType 路由 |
| 新增登录方式要改 Controller | 只需新增实现类，工厂自动发现 |
| 解耦调用者和实现者 | Controller 只依赖接口 |

## 七、常见面试题

### Q1: 为什么使用抽象模板模式？

**A**:
- 各登录方式的核心流程一致（生成 JWT、存 Session）
- 但获取用户的逻辑不同
- 抽象模板模式将"不变"（generateToken）和"变化"（getOrCreateUser）分离

### Q2: 为什么使用工厂模式？

**A**:
- Controller 不需要知道具体实现
- 根据 authType 自动路由到对应实现
- 新增登录方式只需新增实现类，不需要改 Controller

### Q3: Spring 是如何自动注入所有实现的？

**A**:
- Spring 扫描所有 `IAuthService` 的实现类
- 自动注入到 `List<IAuthService>` 中
- 工厂只需遍历 List，根据 authType 筛选

## 八、总结

### 核心认知

| 要点 | 理解 |
|------|------|
| **登录流程** | 工厂路由 → 具体实现 → 抽象模板生成 Token |
| **抽象模板** | 将"不变"（generateToken）和"变化"（getOrCreateUser）分离 |
| **工厂** | 根据 authType 自动路由到对应实现 |
| **扩展性** | 新增登录方式只需新增实现类 |

### 一句话总结

> **工厂负责路由到具体实现，抽象模板负责封装公共逻辑，两者配合实现灵活扩展。**
