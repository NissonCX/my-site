## 二、为什么需要本地数据库？

### 2.1 本地数据库的四大作用

```
┌─────────────────────────────────────────────────────────────────────┐
│                  本地数据库的四大作用                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. 性能优化                                                       │
│     ├─ 问题：每次请求都调用 SSO/用户中心 → 网络开销大、响应慢         │
│     ├─ 解决：登录时同步一次到本地 → 后续查询本地数据库 → 快          │
│     └─ 对比：100ms 网络调用 vs 1ms 本地查询                          │
│                                                                     │
│  2. 高可用性                                                       │
│     ├─ 问题：SSO/用户中心挂了 → 你的系统也挂了？                      │
│     ├─ 解决：本地有数据副本 → 外部系统故障时仍可降级使用              │
│     └─ 策略：外部系统不可用时，返回本地缓存的数据（可能有延迟）        │
│                                                                     │
│  3. 业务扩展                                                       │
│     ├─ 问题：你的业务需要存储额外的数据（如用户偏好、操作记录）        │
│     ├─ 解决：本地数据库可以自由扩展，不依赖外部系统                  │
│     └─ 举例：用户最后登录时间、操作日志、业务配置等                   │
│                                                                     │
│  4. 数据控制                                                       │
│     ├─ 问题：外部系统可能变更接口、限流、收费                         │
│     ├─ 解决：本地有副本 → 不被外部系统"卡脖子"                       │
│     └─ 策略：登录时同步，后续变更通过回调/定时任务更新（如果有）        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 三方副本策略

| 存储位置 | 数据类型 | 读写权限 | 同步方式 |
|---------|---------|---------|---------|
| **阿里 SSO** | 主数据源（基本信息） | 只读（对你） | 登录时读取 |
| **YKZ 用户中心** | 业务数据源（详细信息） | 只读（对你） | 登录时读取 |
| **你的本地数据库** | 数据副本（完整信息） | 读写 | 登录时同步 |

## 三、SSO 系统部署方式

### 3.1 部署架构对比

```
┌─────────────────────────────────────────────────────────────────────┐
│                    SSO 系统部署方式                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【阿里 SSO 系统】（托管在阿里云）                                    │
│     ├─ 部署：阿里云服务器集群（ECS/容器服务）                        │
│     ├─ 数据库：阿里云 RDS/其他云数据库                              │
│     ├─ 协议：OAuth2（互联网标准）                                   │
│     ├─ 优点：                                                       │
│     │  ✅ 高可用（多副本、自动故障切换）                              │
│     │  ✅ 弹性扩展（自动扩容应对流量峰值）                            │
│     │  ✅ 全球部署（CDN 加速）                                       │
│     │  ✅ 运维托管（阿里负责维护）                                   │
│     └─ 缺点：                                                       │
│        ❌ 依赖阿里云（Vendor Lock-in）                             │
│        ❌ 需要网络调用（有延迟）                                     │
│        ❌ 按流量收费（成本不可控）                                   │
│                                                                     │
│  【YKZ 用户中心】（政务专有）                                        │
│     ├─ 部署：政务专有服务器/私有云                                   │
│     ├─ 数据库：政务内网数据库（可能不是云数据库）                     │
│     ├─ 协议：国密 SM2（国家密码标准）                                │
│     ├─ 优点：                                                       │
│     │  ✅ 数据自主可控（符合政务数据安全要求）                        │
│     │  ✅ 安全合规（国密加密，符合国家密码法规）                      │
│     │  ✅ 内网部署（不暴露在公网，更安全）                            │
│     └─ 缺点：                                                       │
│        ❌ 维护成本高（需要自己运维）                                 │
│        ❌ 扩展性受限（硬件资源有限）                                  │
│        ❌ 单点故障风险（没有云厂商的高可用保障）                      │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 四、SSO vs OSS 辨析

### 4.1 概念对比

| 概念 | 全称 | 作用 | 本项目是否使用 |
|------|------|------|--------------|
| **SSO** | Single Sign-On（单点登录） | 身份认证系统 | ✅ 使用 |
| **OSS** | Object Storage Service（对象存储） | 云存储服务（存文件、图片） | ❌ 无关 |

### 4.2 常见误区

```
❌ 错误理解：SSO 数据存在阿里云OSS（对象存储）
✅ 正确理解：SSO 数据存在阿里云数据库（RDS/其他），不是OSS

类比：
  OSS = 云盘（存文件、图片、视频）
  SSO数据库 = 关系型数据库（存用户信息、账号）
```

## 五、数据一致性问题

### 5.1 数据变更场景

```
┌─────────────────────────────────────────────────────────────────────┐
│                   数据一致性的挑战                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  问题场景：                                                         │
│  ─────────────────────────────────────────────────────────────────  │
│  1. 用户在 SSO 系统修改了姓名（张三 → 李四）                          │
│  2. 你的本地数据库还是旧数据（张三）                                  │
│  3. 用户下次登录前，本地数据不一致                                    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 5.2 三种同步方案对比

| 方案 | 策略 | 优点 | 缺点 |
|------|------|------|------|
| **方案1：登录时同步** | 每次登录获取最新数据并覆盖本地 | ✅ 实现简单<br>✅ 不依赖外部回调 | ❌ 有延迟（两次登录之间） |
| **方案2：Webhook 回调** | SSO 数据变更时主动回调 | ✅ 实时更新<br>✅ 一致性好 | ❌ 实现复杂<br>❌ 需要公网接口 |
| **方案3：定时同步** | 每隔一段时间主动同步 | ✅ 可控<br>✅ 不影响登录性能 | ❌ 有延迟 |

### 5.3 当前项目选择

**方案1：登录时同步**

理由：
- 政务场景下，用户信息变更频率低
- 登录时同步即可满足需求
- 实现简单，维护成本低

## 六、常见面试题

### Q1: 为什么不把所有数据存在 SSO？

**A**:
- 分开存储让职责清晰：SSO 负责身份认证，用户中心负责信息管理
- 各自按需选择安全标准：SSO 用 OAuth2，用户中心用国密 SM2
- 数据归属明确

### Q2: 本地数据库有什么作用？

**A**:
- **性能优化**：减少网络调用
- **高可用性**：外部系统故障时仍可降级使用
- **业务扩展**：存储额外数据（用户偏好、操作日志）
- **数据控制**：不被外部系统"卡脖子"

### Q3: SSO 和 OSS 有什么区别？

**A**:
- **SSO**：Single Sign-On（单点登录），身份认证系统
- **OSS**：Object Storage Service（对象存储），存文件、图片
- 两者完全不同，SSO 数据存在数据库，不是 OSS

### Q4: 如何保证数据一致性？

**A**:
- 当前项目选择**登录时同步**
- 每次登录获取最新数据覆盖本地
- 政务场景用户信息变更频率低，这个方案足够

## 七、总结

### 核心认知

| 要点 | 理解 |
|------|------|
| **三方数据分布** | SSO 基本信息 + 用户中心详细信息 = 完整用户数据 |
| **本地数据库作用** | 性能、可用性、业务扩展、数据控制 |
| **SSO vs OSS** | SSO 是单点登录系统，OSS 是对象存储（无关） |
| **数据一致性** | 登录时同步，可扩展 Webhook 或定时同步 |

### 一句话总结

> **SSO 系统存基本信息，用户中心存详细信息，通过 accountId 连接，登录时同步到本地数据库。**
