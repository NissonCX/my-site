## 一、微信登录完整流程

### 1.1 时序图

```
微信小程序          你的后端            微信服务器
   │                  │                    │
   │  1. wx.login()   │                    │
   │  ├─ 获取 code     │                    │
   │  └─ code: "xxx"   │                    │
   │                  │                    │
   │  2. 获取手机号     │                    │
   │  ├─ encryptedData │                    │
   │  └─ iv           │                    │
   │                  │                    │
   │  3. POST /app/login                    │
   │  {code, encryptedData, iv}            │
   │  ├─────────────>│                    │
   │                  │                    │
   │                  │  4. code2session 接口 │
   │                  ├───────────────────>│
   │                  │  {appid, secret, code}
   │                  │                    │
   │                  │  5. 返回 session 信息
   │                  │  {openid, session_key}
   │                  │<───────────────────│
   │                  │                    │
   │                  │  6. 解密手机号
   │                  │  encryptedData + session_key + iv
   │                  │  → phone
   │                  │                    │
   │                  │  7. 创建/绑定用户
   │                  │  查询 openid 是否存在
   │                  │  ├─ 存在 → 返回用户
   │                  │  └─ 不存在 → 创建用户
   │                  │                    │
   │  8. 返回 JWT      │                    │
   │<─────────────┤                    │
   │                  │                    │
```

### 1.2 代码实现

```java
public WeChatSessionResponse getSession(String code) {
    // 调用微信 code2session 接口
    String raw = wechatWebClient.get()
            .uri(uriBuilder -> uriBuilder
                    .path("/sns/jscode2session")
                    .queryParam("appid", weChatConfig.getAppid())
                    .queryParam("secret", weChatConfig.getSecret())
                    .queryParam("js_code", code)
                    .queryParam("grant_type", "authorization_code")
                    .build())
            .retrieve()
            .bodyToMono(String.class)
            .block();

    WeChatSessionResponse response = JSONUtil.toBean(raw, WeChatSessionResponse.class);

    // 错误处理
    if (response.getErrcode() != null && response.getErrcode() != 0) {
        throw new BizException(BizRespCode.ERR_3001);
    }
    return response;
}
```

### 1.3 返回数据结构

```java
public class WeChatSessionResponse {
    private String openid;      // 用户唯一标识
    private String session_key; // 会话密钥
    private String unionid;     // 开放平台唯一标识符（可选）
    private Integer errcode;    // 错误码
    private String errmsg;      // 错误信息
}
```

## 二、核心概念详解

### 2.1 code - 临时登录凭证

```
┌─────────────────────────────────────────────────────────────────────┐
│                       code 的特性                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  生成：                                                            │
│     前端调用 wx.login() → 微信服务器生成 code                        │
│                                                                     │
│  特点：                                                             │
│     ├─ 时效性：5分钟                                               │
│     ├─ 单次性：只能使用一次                                        │
│     └─ 绑定性：绑定当前小程序和用户                                  │
│                                                                     │
│  用途：                                                             │
│     后端用 code + appid + secret 调用微信 API                        │
│     换取 openid + session_key                                       │
│                                                                     │
│  安全设计：                                                         │
│     ├─ 一次性使用 → 防止重放攻击                                   │
│     ├─ 5分钟失效 → 限制攻击窗口                                    │
│     └─ 后端需要 secret → 防止伪造请求                               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 openid vs unionid

```
┌─────────────────────────────────────────────────────────────────────┐
│                   openid vs unionid 对比                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【openid】                                                         │
│     ├─ 作用域：单个小程序                                            │
│     ├─ 特点：同一个用户在不同小程序有不同的 openid                   │
│     └─ 例子：用户A在小程序1的openid = aaa，小程序2的openid = bbb    │
│                                                                     │
│  【unionid】                                                        │
│     ├─ 作用域：微信开放平台下的所有应用                               │
│     ├─ 特点：同一个用户在不同应用有相同的 unionid                   │
│     ├─ 前提：小程序必须绑定到微信开放平台                            │
│     └─ 例子：用户A在所有小程序的 unionid 都是 xxx                   │
│                                                                     │
│  什么时候有 unionid？                                               │
│  ─────────────────────────────────────────────────────────────────  │
│  ✅ 小程序绑定到微信开放平台账号下                                   │
│  ❌ 小程序未绑定开放平台（只有 openid）                               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.3 session_key - 会话密钥

```
┌─────────────────────────────────────────────────────────────────────┐
│                  session_key 的三大作用                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【作用1】解密用户敏感数据                                           │
│  ─────────────────────────────────────────────────────────────────  │
│                                                                     │
│  前端加密的手机号数据：                                              │
│     encryptedData（密文）                                           │
│     iv（初始化向量）                                                │
│                                                                     │
│  后端解密：                                                         │
│     AES-128-CBC 算法                                               │
│     密钥：session_key                                             │
│     明文：encryptedData + session_key + iv → 手机号                  │
│                                                                     │
│  【作用2】校验数据完整性（数字签名）                                  │
│  ─────────────────────────────────────────────────────────────────  │
│                                                                     │
│  场景：前端提交的用户数据需要验证是否被篡改                            │
│                                                                     │
│  【作用3】验证用户身份（会话管理）                                    │
│  ─────────────────────────────────────────────────────────────────  │
│                                                                     │
│  场景：后续请求验证是否是同一个用户                                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.4 session_key 的安全特性

```
┌─────────────────────────────────────────────────────────────────────┐
│                session_key 的安全特性                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. 前端拿不到：                                                    │
│     ✅ session_key 只返回给后端                                     │
│     ✅ 前端无法获取                                                 │
│     ✅ 防止前端泄露                                                 │
│                                                                     │
│  2. 有时效性：                                                      │
│     ✅ 过期后自动失效                                               │
│     ✅ 减少泄露风险                                                 │
│                                                                     │
│  3. 绑定用户：                                                      │
│     ✅ 每个 session_key 绑定特定的 openid                           │
│     ✅ 不能跨用户使用                                               │
│                                                                     │
│  4. 不可逆：                                                        │
│     ✅ 无法从 session_key 推导出 openid                            │
│     ✅ 无法从 session_key 推导出用户信息                            │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 三、IV（初始化向量）

### 3.1 IV 是什么？

```
┌─────────────────────────────────────────────────────────────────────┐
│                      IV 的作用                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  IV = Initialization Vector = 初始化向量                             │
│                                                                     │
│  作用：                                                             │
│     对称加密算法的参数，用于增加加密的随机性                         │
│                                                                     │
│  为什么需要 IV？                                                   │
│  ─────────────────────────────────────────────────────────────────  │
│  ❌ 没有 IV 的问题：                                                │
│     相同的明文 + 相同的密钥 = 相同的密文                              │
│     → 攻击者可以通过模式分析破解                                    │
│                                                                     │
│  ✅ 有 IV 的好处：                                                  │
│     相同的明文 + 相同的密钥 + 不同的 IV = 不同的密文                   │
│     → 增加破解难度                                                 │
│                                                                     │
│  特点：                                                             │
│     ├─ IV 不需要保密，可以公开                                       │
│     ├─ IV 必须是随机的（每次加密都不同）                             │
│     └─ IV 和密文一起传输                                           │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.2 加密解密过程

```
┌─────────────────────────────────────────────────────────────────────┐
│              微信手机号加密/解密完整流程                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【加密阶段】（微信服务器 → 前端）                                   │
│  ─────────────────────────────────────────────────────────────────  │
│                                                                     │
│  微信服务器：                                                        │
│     1. 明文手机号：13800138000                                      │
│     2. 密钥：session_key（临时生成）                                │
│     3. IV：随机生成                                                 │
│     4. 加密算法：AES-128-CBC                                       │
│     5. 密文：encryptedData                                         │
│                                                                     │
│  传输给前端：                                                       │
│     {                                                              │
│       encryptedData: "密文...",                                     │
│       iv: "随机向量"                                                │
│     }                                                              │
│                                                                     │
│  【解密阶段】（后端）                                               │
│  ─────────────────────────────────────────────────────────────────  │
│                                                                     │
│  后端：                                                             │
│     1. 用 code 换取 session_key                                     │
│     2. 用 session_key + iv 解密 encryptedData                      │
│     3. 得到明文手机号：13800138000                                  │
│                                                                     │
│  解密代码：                                                         │
│     WXDecryptUtil.decryptUserInfo(                                 │
│         encryptedData,    // 加密数据                               │
│         session_key,      // 密钥                                  │
│         iv                 // 初始化向量                             │
│     )                                                              │
│     ↓                                                               │
│     {                                                              │
│       "purePhoneNumber": "13800138000",                             │
│       "watermark": {                                               │
│         "timestamp": 1234567890,                                    │
│         "appid": "wx55964d80fa95e9ad"                               │
│       }                                                            │
│     }                                                              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 四、为什么不直接返回手机号？

### 4.1 安全设计

```
┌─────────────────────────────────────────────────────────────────────┐
│              微信手机号获取的安全设计                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ❌ 如果直接返回手机号：                                              │
│     ├─ 前端可以直接拿到手机号                                        │
│     ├─ 恶意小程序可以窃取用户手机号                                  │
│     └─ 隐私泄露                                                     │
│                                                                     │
│  ✅ 使用加密数据 + session_key：                                     │
│     ├─ 前端：encryptedData（加密的手机号）                           │
│     ├─ 后端：用 session_key 解密                                    │
│     ├─ session_key 只在后端，前端无法解密                            │
│     └─ 即使前端被破解，拿到的也是加密数据                             │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 五、微信 vs 渝快办对比

### 5.1 架构差异

| 对比项 | 微信小程序 | 渝快办小程序 |
|-------|----------|------------|
| **JS-SDK** | 内置（无需初始化） | 外挂（需初始化） |
| **初始化 Token** | 无 | jsapiToken |
| **手机号获取** | 前端加密 + 后端 session_key 解密 | 后端直接获取 |
| **API 调用** | 1次 | 1次 |

### 5.2 为什么微信不需要预初始化？

```
┌─────────────────────────────────────────────────────────────────────┐
│              微信 vs 渝快办的架构差异                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【微信小程序】                                                     │
│     ├─ 内置 SDK：微信小程序运行时自带                                │
│     ├─ wx.login()：原生 API，无需初始化                            │
│     └─ 不需要预配置，微信自动处理认证                                │
│                                                                     │
│  【渝快办小程序】                                                   │
│     ├─ 外挂 SDK：第三方 JS-SDK                                      │
│     ├─ 需要手动初始化：YKB.init({jsapiToken, appId, scopes})       │
│     └─ 必须提前获取并配置，否则功能不可用                            │
│                                                                     │
│  类比：                                                            │
│     微信小程序 = iPhone                                            │
│       └─ Face ID 是原生的，随时可用                                  │
│                                                                     │
│     渝快办小程序 = 第三方支付 App                                    │
│       └─ 需要登录、获取 token、配置后才能使用                          │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 六、常见面试题

### Q1: 为什么微信登录比渝快办登录简单？

**A**:
- 微信 API 调用只有 1次，渝快办/渝快政需要 1-5次
- 微信是成熟的标准化平台，政务系统有多方集成

### Q2: session_key 为什么不能暴露给前端？

**A**:
- 前端拿到 session_key 可以解密任意 encryptedData
- 可以伪造用户数据，获取其他用户的敏感信息
- session_key 只在后端保证数据安全

### Q3: code 为什么只能用一次？

**A**:
- 防止重放攻击
- 5分钟后失效，限制攻击窗口
- 每次使用后失效，可以追踪请求来源

## 七、总结

### 核心认知

| 要点 | 理解 |
|------|------|
| **微信登录本质** | OAuth2 授权码模式（简化版） |
| **核心数据** | openid（用户标识）+ session_key（解密密钥） |
| **手机号获取** | 前端加密 + 后端 session_key 解密 |
| **openid vs unionid** | openid 是单应用，unionid 是跨应用 |
| **IV 作用** | 增加加密随机性，防止模式破解 |
| **安全设计** | session_key 不暴露给前端 |

### 一句话总结

> **微信登录流程：前端 wx.login() 获取 code → 后端用 code 换 openid + session_key → 前端获取加密手机号 → 后端用 session_key 解密 → 用手机号创建/查询用户 → 生成 JWT**
