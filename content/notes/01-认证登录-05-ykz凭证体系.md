## 一、四种认证凭证

### 1.1 认证凭证体系

```
┌─────────────────────────────────────────────────────────────────────┐
│                        认证凭证体系                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  【凭证1】code（用户授权凭证）                                        │
│     ├─ 来源：渝快政授权页返回给前端                                  │
│     ├─ 作用：证明"用户已授权"                                       │
│     ├─ 特点：一次性、短时效（5-10分钟）                              │
│     └─ 传输：前端 → 后端                                            │
│                                                                     │
│  【凭证2】ssoAccessToken（应用级 Token）                             │
│     ├─ 获取：appKey + appSecret → 阿里 SSO                         │
│     ├─ 作用：证明"我是合法应用"                                     │
│     ├─ 特点：所有用户共享，应用级凭证                                │
│     └─ 传输：后端 → 阿里 SSO                                        │
│                                                                     │
│  【凭证3】accountId（用户唯一标识）                                  │
│     ├─ 获取：code + ssoAccessToken → 阿里 SSO                      │
│     ├─ 作用：连接 SSO 和用户中心的"桥"                              │
│     ├─ 特点：全局唯一，跨系统一致                                    │
│     └─ 传输：SSO → 后端 → 用户中心                                  │
│                                                                     │
│  【凭证4】userCenterToken（用户中心 Token）                          │
│     ├─ 获取：appId + SM2加密的appSecret → YKZ 用户中心              │
│     ├─ 作用：证明"我有权访问用户中心"                               │
│     ├─ 特点：国密 SM2 加密，政务系统专用                            │
│     └─ 传输：后端 → YKZ 用户中心                                    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 凭证对比

| 凭证 | 来源 | 时效性 | 作用 | 安全考虑 |
|------|------|--------|------|---------|
| **code** | 渝快政授权页 | 5-10分钟 | 证明"用户已授权" | 一次性，防止重放攻击 |
| **ssoAccessToken** | appKey+appSecret换 | 应用级（较长） | 证明"我是合法应用" | 所有用户共享 |
| **accountId** | code+ssoToken换 | 永久 | 连接 SSO 和用户中心 | 全局唯一标识 |
| **userCenterToken** | appId+SM2换 | 较长 | 证明"有权访问用户中心" | 国密加密 |

## 二、accountId 的桥接作用

### 2.1 核心作用

```
┌─────────────────────────────────────────────────────────────────────┐
│                   accountId 的核心作用                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  问题：为什么需要 accountId？                                        │
│                                                                     │
│  答案：                                                             │
│  ─────────────────────────────────────────────────────────────────  │
│  1. 连接两个系统：                                                  │
│     ├─ SSO 系统返回 accountId                                       │
│     ├─ 用户中心用 accountId 查询                                    │
│     └─ 保证两个系统的"同一个人"能关联起来                            │
│                                                                     │
│  2. 全局唯一标识：                                                  │
│     ├─ 同一个用户在不同应用都是同一个 accountId                      │
│     ├─ 不会出现"SSO 说是张三，用户中心说是李四"的情况                │
│     └─ 跨系统数据一致性保证                                         │
│                                                                     │
│  3. 隔离变化：                                                      │
│     ├─ 用户更换手机号 → accountId 不变                              │
│     ├─ 用户调整岗位 → accountId 不变                                │
│     └─ 业务属性变化不影响核心标识                                    │
│                                                                     │
│  类比：                                                            │
│     accountId ≈ 身份证号                                           │
│     ├─ 姓名、手机号、住址都可能变                                    │
│     ├─ 但身份证号永远是你的                                         │
│     └─ 用身份证号保证"是你这个人"                                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 三、openid vs accountId

### 3.1 区别对比

```java
// YkzSsoModel
private String accountId;       // 账号ID
private String openid;          // 应用+用户唯一标识
```

| 字段 | 作用域 | 说明 | 使用场景 |
|------|--------|------|---------|
| **accountId** | 全局 | 同一个用户在不同应用都是同一个 accountId | 跨系统数据关联 |
| **openid** | 应用级 | 同一个用户在不同应用有不同的 openid | 应用内用户标识 |

### 3.2 类比说明

```
accountId = 身份证号（一个人只有一个）
openid    = 各个医院的病历号（同一个人在不同医院有不同编号）

例子：
  用户张三：
    ├─ accountId: 123456（全局唯一）
    ├─ 应用A的openid: aaa111
    ├─ 应用B的openid: bbb222
    └─ 应用C的openid: ccc333
```

## 四、为什么不把所有数据存在 SSO？

### 4.1 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **SSO 存所有数据** | ✅ 只调用一次，简单<br>✅ 延迟更低 | ❌ SSO 和用户中心紧耦合<br>❌ SSO 要升级到国密标准<br>❌ 违反单一职责原则<br>❌ 数据归属不清 |
| **分开存储**（当前） | ✅ 解耦、职责清晰<br>✅ 各自按需选择安全标准<br>✅ 数据归属明确 | ❌ 多一次网络调用 |

### 4.2 结论

对于政务场景，分开存储是合理的选择：
- SSO 系统：身份认证（互联网标准）
- 用户中心：信息管理（国密标准）
- 职责分离，各自按需选择安全标准

## 五、ssoAccessToken 和 userCenterToken 为什么不能共用？

### 5.1 原因分析

```
原因分析：
  1. 系统隔离：
     ├─ SSO 系统和用户中心是两个独立的系统
     ├─ 各自有独立的认证机制
     └─ 不能共用 Token

  2. 安全标准不同：
     ├─ SSO：OAuth2 互联网标准（HTTPS 传输）
     └─ 用户中心：国密 SM2 加密标准

  3. 职责不同：
     ├─ ssoAccessToken：证明"我是合法应用"
     └─ userCenterToken：证明"我有权访问用户中心"

  4. 权限范围不同：
     ├─ ssoAccessToken：只能访问 SSO 接口
     └─ userCenterToken：只能访问用户中心接口
```

## 六、常见面试题

### Q1: 为什么需要 accountId？

**A**:
- 连接 SSO 和用户中心两个系统
- 全局唯一标识，跨系统数据一致性保证
- 隔离变化，业务属性变化不影响核心标识

### Q2: accountId 和 openid 有什么区别？

**A**:
- **accountId**：全局唯一，跨应用一致（类比身份证号）
- **openid**：应用级，同一用户在不同应用有不同 openid（类比病历号）

### Q3: 为什么 SSO 和用户中心不能共用 Token？

**A**:
- 两套独立系统，各自有认证机制
- 安全标准不同（OAuth2 vs 国密 SM2）
- 权限范围不同（只能访问各自的接口）

### Q4: 如果 SSO 系统挂了，会影响用户登录吗？

**A**:
- ❌ 第1步：SSO 认证失败，无法获取 accountId
- ❌ 无法查询用户中心，登录失败
- 降级策略：返回错误提示用户稍后重试

## 七、总结

### 核心认知

| 要点 | 理解 |
|------|------|
| **四种凭证** | code → ssoAccessToken → accountId → userCenterToken |
| **accountId** | 连接 SSO 和用户中心的"桥"，全局唯一标识 |
| **openid vs accountId** | openid 是应用级，accountId 是全局级 |
| **分开存储** | 解耦、职责清晰、各自选择安全标准 |

### 一句话总结

> **accountId 是连接 SSO 和用户中心的桥，保证跨系统数据一致性。**
