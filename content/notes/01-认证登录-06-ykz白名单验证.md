## 一、YkzUserPost 是什么？

### 1.1 数据结构

```java
@Data
public class YkzUserPostDTO {
    private String organizationCode;  // 组织编码
    private Integer postType;         // 岗位类型
}
```

### 1.2 本质含义

**YkzUserPost = 用户的"岗位-组织"关系**

```
一个用户可以有多个岗位：
┌────────────────────────────────────┐
│  用户张三 (accountId=123456)        │
├────────────────────────────────────┤
│  岗位1：第一检察部 (GO_AAAA)       │
│  岗位2：第二检察部 (GO_BBBB)       │
│  岗位3：法警大队     (GO_CCCC)       │
└────────────────────────────────────┘

YkzUserPostDTO 列表：
[
  {organizationCode: "GO_AAAA", postType: 1},
  {organizationCode: "GO_BBBB", postType: 2},
  {organizationCode: "GO_CCCC", postType: 3}
]
```

### 1.3 为什么一个用户有多个岗位？

**政务场景**：
- 公务员可能兼任多个部门职务
- 可能在不同科室有不同岗位
- **只要任一岗位的组织在白名单中，就允许登录**

## 三、关键代码详解

### 3.1 前置判断

```java
if (!ykzConfig.getWhiteUid().contains(ssoModel.getAccountId())) {
    // 只有不在用户白名单的才需要查组织白名单
    List<YkzUserPostDTO> userPostList = listUserPostByAccountId(ssoModel.getAccountId(), accessToken);
    // ...
}
```

**白名单优先级**：
```
1️⃣ 用户白名单（whiteUid）：
   - 配置了 specific accountId（如超级管理员）
   - 直接通过，不查组织

2️⃣ 组织白名单（whiteOrgan）：
   - 普通用户需要检查组织
   - 向上遍历组织树
```

### 3.2 获取岗位列表

```java
List<YkzUserPostDTO> userPostList = listUserPostByAccountId(ssoModel.getAccountId(), accessToken);
```

**HTTP 调用**：
```
你的后端 → YKZ 用户中心
        POST /ykz/user/listUserPostByAccountId
        {
          "data": {"accountId": "123456"},
          "requestId": "uuid"
        }

返回：
[
  {organizationCode: "GO_AAAA", postType: 1},
  {organizationCode: "GO_BBBB", postType: 2}
]
```

### 3.3 遍历岗位

```java
boolean inWhiteList = userPostList.stream()
    .anyMatch(post ->
        hitWhiteOrganUpward(post.getOrganizationCode(), accessToken)
    );
```

**逻辑**：
- 用户可能有多个岗位
- **只要任一岗位的组织在白名单中，就通过**
- 短路求值：找到命中的就停止

### 3.4 向上遍历组织树

```java
private boolean hitWhiteOrganUpward(String organizationCode, String accessToken) {
    String currentOrgCode = organizationCode;

    while (StringUtils.isNotBlank(currentOrgCode)) {
        // 1. 当前组织是否在白名单？
        if (ykzConfig.getWhiteOrgan().contains(currentOrgCode)) {
            return true;  // ✅ 命中！直接返回
        }

        // 2. 查询父组织
        UserOrgan organ = getByOrganizationCode(currentOrgCode, accessToken);
        if (organ == null) {
            return false;  // ❌ 查询失败，返回 false
        }

        // 3. 是否到顶级？
        if (StringUtils.isBlank(organ.getParentOrganizationCode())
                || ykzConfig.getTopParentOrganCode().equals(organ.getParentOrganizationCode())) {
            return false;  // ❌ 到顶了，还没命中
        }

        // 4. 继续向上
        currentOrgCode = organ.getParentOrganizationCode();
    }

    return false;
}
```

**调用示例**：
```
用户所在：GO_AAAA（第一检察部）

第1次循环：
  ├─ currentOrgCode = "GO_AAAA"
  ├─ 检查白名单 → ❌
  ├─ HTTP: 查询 GO_AAAA → 返回父组织 GO_BBBB
  └─ 继续

第2次循环：
  ├─ currentOrgCode = "GO_BBBB"
  ├─ 检查白名单 → ✅ 是！
  └─ return true（停止遍历）
```

### 3.5 查询单个组织

```java
public UserOrgan getByOrganizationCode(String organizationCode, String accessToken) {
    JSONObject param = new JSONObject()
        .fluentPut("data", new JSONObject()
            .fluentPut("organizationCode", organizationCode)
        ).fluentPut("requestId", IdUtil.randomUUID());
    Map<String, String> headers = new HashMap<String, String>(1) {{
        put("Authorization", "Bearer " + accessToken);
    };

    YkzBaseRespModel resp = doJsonPost(
        ykzConfig.getUserCenter().getPath() + "/ykz/org/getByOrganizationCode",
        param,
        headers
    );

    return JSONObject.parseObject(resp.getData()).toJavaObject(UserOrgan.class);
}
```

**UserOrgan 数据结构**：
```java
@Data
public class UserOrgan {
    private String organizationCode;        // 组织编码（关键字段）
    private String parentOrganizationCode;  // 父组织编码（关键字段）
    private String name;                    // 组织名称
    private String orgType;                 // 组织类型
    private String govInstitutionLevelCode; // 机构级别
}
```

## 四、为什么这样设计？

### 4.1 政务系统的组织结构

```
重庆市检察院系统（树状结构）
│
├─ 市检察院（省级）
│     │
│     ├─ 第一分院
│     ├─ 第二分院
│     └─ ...
│
├─ 九龙坡区人民检察院（区级）
│     │
│     ├─ 第一检察部
│     ├─ 第二检察部
│     ├─ 第三检察部
│     ├─ 法警大队
│     └─ 办公室
│
└─ ...
```

**业务需求**：
- 系统只给"九龙坡区人民检察院"的人用
- 用户可能属于任何下级部门
- 新增部门时不想改配置

**设计选择**：向上遍历组织树

### 4.2 两种配置方案对比

| 方案 | 配置方式 | 优点 | 缺点 |
|------|---------|------|------|
| **精确配置** | 把所有部门都写进白名单 | 权限精确 | 维护成本高 |
| **向上遍历** | 只配置上级组织 | 维护成本低 | 可能授权过宽 |

**当前项目选择**：向上遍历

**理由**：
- 政务场景下，子组织继承父组织权限是合理的
- "第一检察部的人肯定是检察院的人"
- 降低维护成本（新增部门不需要改配置）

## 五、whiteUid vs whiteOrgan 的分工

### 5.1 whiteUid：特殊人员

```yaml
whiteUid:
  - 491727  # 系统管理员
  - 70885   # 技术负责人
```

**作用**：
- 这些人**不属于任何检察院**，但需要登录系统
- 可能是开发人员、运维人员、政务局的管理人员
- 直接放行，不查组织

### 5.2 whiteOrgan：普通用户

```yaml
whiteOrgan:
  - GO_1607305640156467200  # 九龙坡区人民检察院
```

**作用**：
- 普通公务员必须属于某个组织
- 通过组织控制访问权限
- 向上遍历是合理的

## 七、常见面试题

### Q1: 为什么要向上遍历组织树？

**A**:
- **降低维护成本**：只配置上级组织，所有下级部门自动有权限
- **符合政务场景**：子组织继承父组织权限是合理的
- **应对组织变化**：新增部门、部门改名不需要改配置

### Q2: whiteUid 和 whiteOrgan 的优先级？

**A**:
- **whiteUid 优先**：先检查用户白名单，命中则直接通过
- **whiteOrgan 兜底**：普通用户检查组织白名单

### Q3: 如果用户有多个岗位，怎么处理？

**A**:
- 遍历所有岗位
- **只要任一岗位的组织在白名单中，就通过**
- 使用 `anyMatch` 短路求值

### Q4: 为什么 HTTP 失败返回 false？

**A**:
- **优先保证安全性**：查不到组织 = 不能保证身份 = 不能允许登录
- **宁可拒绝合法用户，不能放行非法用户**
- **YKZ 稳定性高**：政务系统，故障概率低，这个 trade-off 可接受

## 八、总结

### 核心认知

| 要点 | 理解 |
|------|------|
| **YkzUserPost** | 用户的"岗位-组织"关系，一个用户可有多个岗位 |
| **白名单优先级** | whiteUid > whiteOrgan |
| **向上遍历** | 降低维护成本，子组织继承父组织权限 |
| **遍历策略** | 任一岗位命中白名单即可通过 |
| **设计合理性** | 符合政务场景，YKZ 是可信的政务系统 |

### 一句话总结

> **whiteUid 给特殊人员用，whiteOrgan 给普通用户用**
> **向上遍历是为了降低维护成本**
> **只要任一岗位的组织在白名单中就通过**
> **优先保证安全性，HTTP 失败则拒绝登录**
